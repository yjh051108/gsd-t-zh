#### Goal
Make a two-altitude, intention-first PseudoCode document the signed-off source-of-truth at the FRONT of every milestone, and wire it MECHANICALLY downstream so partition/plan trace to it and verify checks the build against it — a guard-map divergence is a FAILURE at contract severity. When the behavior map is confirmed in language before code, partition→plan→execute→verify runs autonomously and delivers exactly what's expected — eliminating the 50–100-turn course-corrections measured twice this month.

#### Risk-First Framing (the distinct angle of this proposal)
The whole milestone's value rests on ONE uncertain claim: **that a prose `[RULE]` guard map can be turned into machine-checkable test assertions where divergence is a deterministic, non-vacuous FAILURE.** Everything else (template, two-altitude flow, sign-off gate, doc-ripple) is low-risk authoring/wiring work GSD-T has done many times. So this milestone is sequenced to PROVE or KILL the load-bearing claim before spending on the easy parts. If the guard→assertion bridge can't be made deterministic, we want to know on the first wave — not after building the template and the sign-off UX around a hollow center.

**Risk-ranked work order (highest uncertainty first):**

1. **R1 — Guard-map → assertion bridge (HIGHEST RISK, prove first).** Can a `[RULE]` line in a PseudoCode doc be parsed and bound to a verify-time check that FAILS on divergence and does NOT pass vacuously? This is novel — the existing traceability-gate (`bin/gsd-t-traceability-gate.cjs`, M83) binds ACs→(code path + named test) but never reads a prose spec doc or judges intention coverage. The new mechanism must: (a) enumerate every `[RULE]` from `PseudoCode-[Title].md`, (b) detect that each rule maps to ≥1 test assertion, (c) fail when a rule is unbacked OR contradicted (a divergence), at the same severity as a contract breach. The two binvoice exemplars are the fixture corpus — if the bridge can't deterministically extract their guard maps and prove a planted divergence fails, the milestone's thesis is false. **Spike this in wave 1 against both exemplar docs before any template or command work.** Falsifiable kill-criterion: a `[RULE]`-map parser + verify check that, on a doctored exemplar with one rule deliberately violated by the "build," exits non-zero; and on the faithful build, exits zero — both deterministically, zero LLM-judgment in the pass/fail decision (LLM may map, code must gate).

2. **R2 — Traceability coupling: section-coverage + the competition-altitude shift (HIGH RISK, prove second).** Plan tasks must CITE the pseudocode section they implement (a section with no task = coverage gap, like journey-coverage); the M83 traceability-gate extends to require this citation. Risk: the existing gate is purely tasks.md-internal — adding a cross-document coverage check (tasks.md ↔ PseudoCode-[Title].md sections) is new surface and the place a second non-converging-Red-Team loop could ignite (cf. [[feedback_coverage_check_structural_not_substring]] — coverage must parse structure, never substring-match). Also at risk: backlog #34 says competition (M82) shifts UP to the high-level-approach altitude when behavior is spec'd — that interaction must be specified, not discovered at integrate. Prove the section-coverage check is structural and the competition-altitude rule is decided before building the authoring flow.

3. **R3 — Keep-or-supersede guardrail + divergence provenance (MEDIUM RISK).** Before encoding ANY model inherited from shipped code, the agent MUST ask "keep this or supersede it?" per inherited model; every supersede emits a `⚠ Divergence` flag. This is the exact failure from the PayPal exchange (agent over-trusted the shipped S2-M4 stored-draft model; user: "We shouldn't be saving any draft of anything"). Medium risk because it's a prompt-protocol obligation, not a deterministic gate — its reliability is bounded by how forcing we can make the question. 4. **R4 — Two-altitude milestone flow + sign-off gate (LOW RISK, build last).** High-level pseudocode first (approach: what/why/when, actors, one-breath summary) → user signs off the APPROACH → detailed decomposition to `PseudoCode-[Title].md` at PayPal-doc granularity → milestone is NOT "DEFINED" until the detailed doc is signed off. Default-ON for all milestones (greenfield benefits most — no shipped code to ground in, so the pseudocode IS the spec); skippable per-milestone for trivial work, skip is a LOGGED decision not silent ([[feedback_no_silent_degradation]]). Low risk: this is `/gsd-t-milestone` + its phase-workflow flow plus a sign-off checkpoint — known GSD-T patterns.

5. **R5 — Template + doc-ripple integration (LOWEST RISK, build last).** Ship `templates/PseudoCode-spec.md` (intention/mechanism/guard-map/divergence/appendix skeleton + two-altitude structure), anchored to the binvoice exemplars not a blank page. Add `PseudoCode-[Title].md` to the Living Documents ripple set: `templates/CLAUDE-global.md` Living Documents table (line ~60), Pre-Commit Gate, `commands/gsd-t-doc-ripple.md` — so when code/contract/schema changes mid-milestone, the pseudocode doc updates in the same pass and stays the truth, not a stale snapshot.

#### Document format (observed, from the two binvoice exemplars)
Each section pairs a `> **Intention ({Author}, {date}).**` prose block (the WHY/directive — dated + attributed for durable provenance) ABOVE a fenced pseudocode block (the verified HOW), plus: a "one call in one breath" summary table, a guard-enumeration map rendering every invariant as a one-line `[RULE]`, explicit `⚠ Divergence` flags wherever a new intention supersedes shipped code, and a raw-pseudocode appendix with intention prose stripped (the build's quick-reference). Grounds in EXISTING contracts/schema rather than inventing field names; defers concrete identifiers to plan-time-against-real-schema. Doc prose is INTENTION (the user's), never agent reasoning — agent reasoning stays out of the doc.

**Naming convention.** Instances are `PseudoCode-[Title].md` where `[Title]` names what the doc REPRESENTS (its subject), NOT a milestone id — e.g. `PseudoCode-PayPal.md`, `PseudoCode-Extension.md` (both binvoice exemplars coexist today). A project holds MANY pseudocode docs, and a single milestone may produce several (one per coherent subject/subsystem). Only `templates/PseudoCode-spec.md` keeps the `-spec` name — it is the shipped blank skeleton (the mold), not an instance.
