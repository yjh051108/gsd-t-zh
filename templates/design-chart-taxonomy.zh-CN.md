# 设计图表与原子分类法（封闭集合）

将设计分解为元素合同时，必须从此枚举列表中选择。**不要自创元素名称。** 如果设计元素不符合其中任何一个，停止并请用户在继续之前扩展分类法。

## 为什么这是封闭集合

灾难性故障模式：agent 在 Figma 中看到"条形"，选择了 `chart-bar-grouped-vertical`，但设计实际上是 `chart-bar-stacked-horizontal-percentage`。不同的元素，不同的合同，不同的数据绑定。封闭枚举配合视觉区分器可防止这种情况。

## 图表

### 条形图

| 元素名称                                   | 视觉区分器                                                                 |
|------------------------------------------------|--------------------------------------------------------------------------------------|
| `chart-bar-horizontal-single`                  | 每类一个条形，水平方向，单系列                          |
| `chart-bar-vertical-single`                    | 每类一个条形，垂直方向，单系列                            |
| `chart-bar-grouped-horizontal`                 | 每类多个条形**并排**，水平方向，≥2 个系列（需要图例）     |
| `chart-bar-grouped-vertical`                   | 每类多个条形**并排**，垂直方向，≥2 个系列（需要图例）       |
| `chart-bar-stacked-horizontal`                 | 每类一个条形内**堆叠**分段，水平方向，≥2 个系列且为绝对值 |
| `chart-bar-stacked-vertical`                   | 每类一个条形内**堆叠**分段，垂直方向，≥2 个系列且为绝对值   |
| `chart-bar-stacked-horizontal-percentage`      | **单个**水平条形，100% 宽度，分段总和为 100%（分布可视化）           |
| `chart-bar-stacked-vertical-percentage`        | **单个**垂直条形，100% 高度，分段总和为 100%                               |
| `chart-bar-diverging-horizontal`               | 条形从中心轴向**左右**双向延伸（情感, 正/负）                   |
| `chart-bar-range-horizontal`                   | 浮动条形显示最大-最小范围（无原点轴）                                 |
| `chart-bar-waterfall-vertical`                 | 连续条形显示累积变化，正/负                                   |

**条形图决策规则:**
1. 图表是否显示分段总和为 100% 的**一个**条形？ → `*-stacked-*-percentage`
2. 图表是否显示多个条形堆叠在一起？ → `*-stacked-*`（绝对值时非百分比）
3. 图表是否显示每类多个并排条形？ → `*-grouped-*`
4. 单系列，每类一个条形？ → `*-single`

| 元素名称              | 视觉区分器                                      |
|---------------------------|-----------------------------------------------------------|
| `chart-pie`               | 完整圆形，无孔                                      |
| `chart-donut`             | 带中心孔的圆，可显示中心标签/数值      |

## 分解过程中使用此分类法

当 `gsd-t-design-decompose` 运行时，对于设计中遇到的**每个视觉元素**：

1. **确定类别**（图表 / 图例 / 坐标轴 / 卡片 / 表格 / 控件 / 原子 / 排版 / 布局）
2. **从此文件中选择精确的元素名称** — 不要重命名，不要缩写
3. **如果未找到匹配** → 停止。向用户报告："位于 Figma 节点 {nodeId} 的元素与任何分类法条目不匹配。建议的新变体：{名称}，原因：{理由}。我应该扩展分类法吗？"
4. **永远不要选择近似匹配** — "差不多" 正是水平堆叠条形百分比变成垂直分组条形的原因。

### 模糊案例的视觉区分器

**"这是堆叠条形图还是分组条形图？"**
- 单个类别的条形**接触/堆叠**还是**并排**？
  - 接触/堆叠 → `chart-bar-stacked-*`
  - 并排 → `chart-bar-grouped-*`

**"这是堆叠绝对值还是堆叠百分比？"**
- 分段是否填充**固定宽度/高度**，无论数据如何？
  - 是（所有条形长度相同） → `*-stacked-*-percentage`
  - 否（条形长度按总值变化） → `*-stacked-*`（非百分比）

**"这是饼图还是环形图？"**
- 中心有孔吗？
  - 是 → `chart-donut`
  - 否 → `chart-pie`

**"这是条形图还是直方图？"**
- x 轴值是**分类**（不同标签）还是**分箱**（数值范围）？
  - 分类 → `chart-bar-*`
  - 数值分箱 → `chart-histogram`

## 命名语法

元素名称遵循一致的语法以避免即兴发明：

```
{category}-{variant}-{orientation/modifier}

类别: chart, axis, legend, stat-card, card, table, list, button, input,
           select, tabs, filter, icon, text, heading, container, stack, grid

修饰符（常用）:
  -single / -multi           — 系列数量
  -horizontal / -vertical    — 方向
  -stacked / -grouped        — 排列方式
  -percentage                — 值总和为 100%
  -dense / -comfortable      — 密度
  -outline / -filled         — 视觉重量
```

提出新条目时，匹配其同级部分的语法。

## 扩展分类法

如果需要新的元素变体：

1. **先写提案**：`.gsd-t/contracts/design/taxonomy-proposals/{name}.proposal.md`，包含：
   - 它所属的部分（现有或新）
   - 建议的条目名称 + 视觉区分器（≤80 字符）
   - 同级条目 — 新条目与每个有何不同
   - 灾难性错误分类理由（如果将其塞入现有条目会出什么问题）
   - 来源（Figma 节点, 截图）
   - 关联条目（如果新条目意味着一组同级条目）
2. 将其添加到相关部分并附带视觉区分器
3. 如果创建新部分，将其放在结构相似的部分附近（布局类靠近表格，交互类靠近控件）
4. 如果是新的图表变体，在图表类型下添加决策规则
5. 提升 GSD-T 版本（次要）并在 CHANGELOG.md 中记录添加内容
6. 永远不要删除条目 — 只添加（破坏封闭集合是破坏性操作）
