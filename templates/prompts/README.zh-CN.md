# GSD-T 子代理提示模板

此目录保存曾经内联到命令 markdown 文件中的长格式提示。内联它们会导致大量的上下文消耗 — 每个 Task 子代理生成会重新生成约 3000 个令牌的提示模板，每个里程碑重复数十次。

现在命令文件通过路径引用它们。编排器将文件路径传递给子代理，子代理自行读取。编排器永远不会在自己的上下文中持有完整提示。

## 文件

| 文件 | 用途 | 运行频率 |
|------|---------|---------------|
| `qa-subagent.md` | 测试生成、执行、差距报告 | 每个任务 |
| `red-team-subagent.md` | 对抗性缺陷搜寻 | 每个域（不是每个任务）|
| `design-verify-subagent.md` | 逐元素视觉设计审计 | 每个域（不是每个任务）|

## 为什么是 per-domain 而不是 per-task

Red Team 和设计验证最初是按域运行的。它们通过提交 `da6d3ae` 和 `b68353e` 被提升为按任务运行，基于一个假设：编排器的基于环境变量的上下文自检会在上下文耗尽变得严重之前捕获它。该环境变量路径从未被 Claude Code 填充 — 自检是空谈。在它失效的情况下，每次生成约 10k 令牌的 Red Team 子代理在 5-10 个任务内耗尽会话。将它们恢复为 per-domain 将安全任务计数从约 5 提高到 15+。v2.0.0（M34）用通过上下文仪表盘 PostToolUse 钩子的真实 `count_tokens` 测量取代了破损的检查。

QA 保持 per-task 因为（a）它小得多，（b）它根据可能逐任务漂移的合同进行基础验证。

## 添加新提示

将提示作为独立的 markdown 文件写入此目录。从命令文件中通过以下方式引用它：

```
生成 Task 子代理（general-purpose，模型: <模型>）:
"读取 `templates/prompts/<your-prompt>.md` 并遵循它。本次运行的上下文: <一行上下文>。"
```

将内联上下文保持在一行。提示正文必须存在于文件中，而不是命令中。
