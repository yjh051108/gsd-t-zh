# 安全标准（通用 — 所有项目）

这些规则是 **强制** 的。违规导致任务失败。没有例外。

---

## 1. 输入验证与清理

### 所有用户输入
- **在系统边界验证** — 来自用户、外部 API、URL 参数、表单字段、文件上传的每个值
- **白名单，不要黑名单** — 定义什么是 **允许的**，拒绝其他所有内容
- **在处理之前验证类型、长度、格式和范围**
- **在处理前修剪和规范化** 字符串（Unicode 规范化、空白）

### SQL 注入防护
```
强制:
  ├── 始终使用参数化查询 / 预编译语句
  ├── 绝不将用户输入拼接成 SQL 字符串
  ├── 优先使用 ORM 查询构建器而非原始 SQL
  └── 如果原始 SQL 不可避免，使用 ORM 的参数化转义
```

### 命令注入防护
```
强制:
  ├── 绝不将用户输入传递给 shell 命令（exec, spawn, system）
  ├── 如果 shell 执行不可避免，仅使用允许列表中的命令
  ├── 使用 execFile（不是 exec）与显式参数数组
  └── 绝不使用来自用户数据的字符串插值构建命令
```

## 2. 跨站脚本（XSS）

### 输出编码
- **在渲染到 HTML、JavaScript、CSS 或 URL 之前编码所有动态输出**
- **上下文感知编码** — HTML 中的 HTML 实体、script 标签中的 JS 转义、URL 中的 URL 编码
- **绝不信任任何来源的数据** — 甚至是你自己的数据库（它可能包含先前注入的内容）

### DOM 操作
```
强制:
  ├── 绝不在用户数据上使用 innerHTML, outerHTML, 或 document.write
  ├── 使用 textContent 或 innerText 进行文本插入
  ├── 如果需要 HTML 渲染，先用 DOMPurify 清理
  │     允许的标签: ['p', 'br', 'strong', 'em', 'ul', 'ol', 'li', 'a', 'b', 'i', 'u']
  │     允许的属性: ['href']（仅在 <a> 上，带 URL 验证）
  ├── React: 绝不在没有 DOMPurify 的情况下使用 dangerouslySetInnerHTML
  └── Vue: 绝不在没有 DOMPurify 的情况下使用 v-html
```

### 内容安全策略
- 在所有响应上设置 `Content-Security-Policy` 头
- 最低限度: `default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'`
- 绝不在生产中允许 `'unsafe-eval'`

## 3. 认证与会话管理

### Token 处理
```
强制:
  ├── 绝不在 localStorage 或 sessionStorage 中存储认证令牌
  │     （任何 XSS 攻击都可以访问 — 游戏结束）
  ├── 对会话令牌使用 httpOnly、Secure、SameSite cookies
  ├── 对于 SPAs: 将令牌存储在内存中（React Context、闭包）
  │     接受页面刷新 = 重新认证
  ├── 访问令牌: 短有效期（最多 15 分钟）
  ├── 刷新令牌: 仅 httpOnly cookie，使用时轮换
  └── 绝不在 URL 或查询参数中包含令牌
```

### 密码处理
- **绝不明文存储密码** — 使用 bcrypt、scrypt 或 Argon2
- **最少 12 个字符要求** — 无最大长度限制
- **绝不记录密码** — 即使哈希过的也不行
- **限制登录尝试** — 5 次失败 → 渐进延迟或锁定

### 会话安全
- 认证后重新生成会话 ID
- 设置会话超时（空闲: 30 分钟，绝对: 8 小时）
- 密码更改时使所有会话失效

## 4. API 安全

### 请求验证
```
强制:
  ├── 验证 Content-Type 头匹配预期格式
  ├── 使用 schema 验证请求体（Zod, Joi, JSON Schema）
  ├── 拒绝包含意外字段的请求（additionalProperties: false）
  ├── 设置最大请求体大小（默认: 1MB）
  └── 限制所有端点的速率（API 默认: 每分钟每 IP 100 个请求）
```

### 响应安全
- 绝不在生产响应中包含堆栈跟踪或内部错误详情
- 对认证失败使用通用错误消息（"无效凭证" — 不要揭示哪个字段失败了）
- 在所有响应上设置安全头:
  - `X-Content-Type-Options: nosniff`
  - `X-Frame-Options: DENY`
  - `Strict-Transport-Security: max-age=31536000; includeSubDomains`
  - `X-XSS-Protection: 0`（依赖 CSP 替代）

### CORS
- 绝不在认证端点上使用 `Access-Control-Allow-Origin: *`
- 将特定来源列入白名单
- 绝不在不验证的情况下将 Origin 头反映为 CORS 来源

## 5. 密钥管理

```
强制:
  ├── 绝不在源代码中硬编码密钥（API 密钥、密码、令牌）
  ├── 绝不提交包含真实密钥的 .env 文件
  ├── 对所有密钥使用环境变量
  ├── 绝不在日志中记录密钥 — 在所有日志输出中脱敏
  ├── 绝不在前端/客户端代码中包含密钥
  │     （所有客户端代码对用户可见）
  ├── 绝不在错误消息或堆栈跟踪中包含密钥
  └── 定期并在任何 suspected 泄露时轮换密钥
```

### .gitignore 要求
这些文件必须始终在 `.gitignore` 中:
- `.env`, `.env.local`, `.env.*.local`
- `*.pem`, `*.key`, `*.cert`
- `credentials.json`, `service-account.json`
- `*.sqlite`, `*.db`（本地数据库）

## 6. AI 特定安全（LLM / 提示词注入）

### 提示词注入防护
```
强制（当用户输入被输入到 LLM 调用时）:
  ├── 绝不将用户输入直接拼接成系统提示词
  ├── 使用结构化消息格式: 系统消息（可信）+ 用户消息（不可信）
  ├── 在将用户输入包含到任何提示词之前验证和清理
  ├── 去除或转义控制序列（markdown、XML 标签、类似指令的文本）
  ├── 设置输出长度限制以防止提取攻击
  ├── 记录和监控提示词输入中的注入尝试
  └── 绝不允许用户输入修改系统指令或工具定义
```

### LLM 输出处理
- **绝不信任 LLM 输出** — 将其视为不可信输入
- 执行前验证 LLM 生成的代码
- 渲染前清理 LLM 生成的 HTML/markdown
- 绝不在不验证的情况下执行 LLM 建议的 shell 命令
- 如果 LLM 输出输入到数据库查询，使用参数化查询

### 提示词中的敏感数据
- 绝不在 LLM 提示词中包含 PII、密码或 API 密钥
- 在发送到外部 LLM API 之前脱敏敏感字段
- 意识到 LLM API 提供者可能记录提示词 — 将所有提示词内容视为可能被记录

## 7. 文件上传安全

```
强制（处理文件上传时）:
  ├── 通过内容（magic bytes）而不仅是扩展名验证文件类型
  ├── 设置最大文件大小限制
  ├── 生成随机文件名 — 存储时永远不要使用原始文件名
  ├── 将上传存储在 web 根目录之外
  ├── 处理前扫描恶意软件
  ├── 绝不要执行或解释上传的文件
  └── 文件下载时设置 Content-Disposition: attachment
```

## 8. 依赖安全

- **定期审计依赖** — `npm audit`, `pip audit`, `go mod verify`
- **固定依赖版本** — 使用锁文件（package-lock.json, poetry.lock）
- **绝不要从不信任的来源安装包**
- **审查依赖许可证** 的兼容性
- **监控生产依赖中的 CVE**

---

## 9. 日志与错误处理

### 要记录的内容
- 认证事件（登录、登出、失败尝试）
- 授权失败（访问被拒绝）
- 输入验证失败
- 应用错误和异常
- 安全相关事件（速率限制、CORS 拒绝）

### 绝不记录的内容
- 密码（即使哈希过的）
- 会话令牌或 API 密钥
- 完整信用卡号
- 社会安全号码或政府 ID
- 个人健康信息
- 任何受 PII 法规约束的数据

### 错误处理
- 绝不在生产中向用户暴露堆栈跟踪
- 使用带错误代码的结构化错误响应
- 服务端记录完整错误，向客户端返回清理版本
- 绝不捕获并静默吞掉错误 — 至少记录它们

---

## 10. 外部链接

```
强制:
  ├── 所有外部链接: target="_blank" rel="noopener noreferrer"
  ├── 重定向前验证 URL（防止开放重定向攻击）
  ├── 绝不在不验证的情况下重定向到用户提供的 URL
  └── 将允许的重定向域列入白名单
```

## 提交前安全检查清单

提交处理用户输入、认证或外部数据的代码之前:

- [ ] 源代码中没有硬编码密钥
- [ ] 用户输入在系统边界被验证和清理
- [ ] SQL 查询使用参数化语句
- [ ] 没有未经清理的 `innerHTML` / `dangerouslySetInnerHTML`
- [ ] 认证令牌未存储在 localStorage 中
- [ ] 错误响应不暴露内部详情
- [ ] 外部链接使用 `rel="noopener noreferrer"`
- [ ] 文件上传按内容类型和大小验证
- [ ] 敏感数据未被记录
- [ ] 如果集成了 LLM: 已实施提示词注入防护
